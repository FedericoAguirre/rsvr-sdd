# Research: Translate English Tags into Spanish

## Overview

Research into the project's internationalization (i18n) setup and identification of untranslated English strings in the `clients/search` and `admin/equipment/equipment/` endpoints.

## Findings

### Django i18n Configuration

The project has a fully functional i18n system:

- **LocaleMiddleware** is installed in `backend/config/settings.py:27`
- **LANGUAGE_CODE** is set to `"es"` (Spanish is the default and only language)
- **LANGUAGES** defined as `[("es", "Español")]`
- **LOCALE_PATHS** points to `backend/locale/`
- **USE_I18N** and **USE_L10N** are both `True`

**Conclusion**: The i18n infrastructure is complete. No configuration changes needed.

### Translation File Structure

The translation file at `backend/locale/es/LC_MESSAGES/django.po` follows standard GNU gettext format:

```po
msgid "English Source String"
msgstr "Spanish Translation"
```

For strings with template variables:

```po
#, python-format
msgid "%(client)s – %(equipment)s – %(date)s"
msgstr "%(client)s – %(equipment)s – %(date)s"
```

The compiled `.mo` file is at the same location. Running `django-admin compilemessages` regenerates it.

### Equipment Admin — Root Cause

The `admin/equipment/equipment/` strings `"Seleccione equipment a modificar"` and `"AÑADIR EQUIPMENT"` are generated by Django admin using the model's `verbose_name`. The `Equipment` model (`backend/apps/equipment/models.py:28`) currently sets only `verbose_name_plural = _("equipment")` but has **no singular `verbose_name`**. Django falls back to the class name `"Equipment"` (untranslated).

**Fix**: Add `verbose_name = _("Equipment")` to the Equipment model's `Meta` class.

### Template i18n Patterns

All existing templates use Django's standard i18n template tags:

- `{% load i18n %}` — load the i18n tag library
- `{% translate "String" %}` — simple static string translation
- `{% blocktranslate %}String with {{ variable }}{% endblocktranslate %}` — strings with template variables
- `_("String")` — in Python code (forms, models, views)

The `clients/search.html` and `clients/_search_results.html` templates already use these tags for the untranslated strings. They just need entries added to the PO file.

### JavaScript Translation

All affected strings are in Django templates or Python forms. No JavaScript-based translation is required.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Translation mechanism | Django i18n `.po`/`.mo` files | Already in use project-wide; no new tooling needed |
| Equipment verbose_name | Add `verbose_name = _("Equipment")` | Fixes both admin display strings in one change |
| Singular vs plural "Equipment" | `msgstr "Equipo"` for singular, `msgstr "equipos"` for plural | Context-appropriate Spanish; admin uses singular, lists use plural |
| Translation workflow | Edit `.po` → compile `.mo` | Standard Django workflow |

## Alternatives Considered

- **django-rosetta**: Web-based translation UI. Rejected — not installed as a dependency, adds unnecessary complexity for 8 strings.
- **Inline JavaScript replacement**: Translating in JS client-side. Rejected — inconsistent with existing i18n approach.
- **Translation memory/automation**: Using LLM to batch-generate translations. Could be used but manual review still required for context accuracy.
